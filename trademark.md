# OpenTT Trademark and Brand Usage Policy

## 1. Purpose

OpenTT is free and open software created to support the digitalization and long-term preservation of table tennis infrastructure.
The source code is licensed under GNU AGPL terms, which grant broad rights to use, study, modify, and share the software.

This document covers a separate topic: use of the OpenTT name, logo, and brand identity.

Its purpose is not to restrict the community, but to protect project integrity, user trust, and clarity around what is official.

## 2. Scope

This policy applies to:
- the name `OpenTT`,
- OpenTT logos and visual marks,
- official project identity and presentation,
- statements implying official affiliation, endorsement, or stewardship.

It does not limit AGPL software freedoms in source code usage. Code freedoms and brand usage are distinct legal/operational layers.

## 3. Open Source Code Rights (Code License Layer)

Under AGPL terms, you may:
- use OpenTT code,
- study and audit OpenTT code,
- modify OpenTT code,
- self-host OpenTT instances,
- fork and redistribute in compliance with AGPL obligations.

If you provide modified OpenTT as a networked service, AGPL source-sharing obligations apply.

## 4. Brand and Identity Rules (Trademark Layer)

Even when code use is allowed, you must not use OpenTT branding in ways that can mislead users.

### 4.1 Allowed Without Prior Permission

- truthful references to OpenTT as the upstream project,
- factual statements such as "based on OpenTT" or "forked from OpenTT",
- linking to the official repository and documentation,
- discussing compatibility with OpenTT,
- community contributions and public technical discussion with accurate attribution.

### 4.2 Not Allowed Without Prior Permission

- presenting an unofficial build/fork as the official OpenTT project,
- using the OpenTT name/logo in a way likely to confuse users about authorship or endorsement,
- claiming formal partnership, certification, or approval that does not exist,
- imitating official infrastructure, release channels, or support identity,
- using OpenTT branding as the primary product identity for a divergent fork without clear distinction.

## 5. Required Attribution for Forks and Derivatives

If you publish a fork or derivative distribution, you should:
- clearly identify it as unofficial (for example: "Unofficial fork of OpenTT"),
- provide a visible link to the upstream OpenTT project,
- avoid naming/branding choices that imply your version is the canonical upstream.

Recommended wording:
- "This project is an independent fork based on OpenTT."
- "Not affiliated with or endorsed by the official OpenTT project."

## 6. Domain, Social, and Distribution Naming Guidance

To reduce user confusion, avoid:
- domain names that look like official OpenTT infrastructure unless you are the official maintainer,
- social handles or package names that imply canonical ownership.

Prefer explicit derivative naming, for example:
- `<your-org>-opentt`,
- `opentt-<region>-fork`,
- `<product-name> (based on OpenTT)`.

## 7. Logo and Visual Identity Usage

Do not use official OpenTT logos/marks as the primary identity of a modified distribution unless authorized.
When showing compatibility or origin, logo usage must be secondary and clearly contextual.

If in doubt, use plain-text attribution instead of logo usage.

## 8. Official vs Unofficial Communication

Only official project channels may publish statements as "OpenTT official."
Fork maintainers and integrators must communicate in their own name and entity identity.

Users should always be able to understand:
- who operates the service/software they are using,
- whether it is official upstream OpenTT or a third-party derivative.

## 9. Why These Rules Exist

These rules exist to protect:
- data trust and operational reliability,
- users from accidental confusion,
- community confidence in upstream releases,
- long-term sustainability of an open public-interest software project.

The policy is designed to preserve openness while preventing identity misuse.

## 10. In-Progress Modules and Brand Representation

Some bundled modules (for example, `addons/tournaments`) may be in-progress and not yet feature-complete.
Do not market incomplete bundled modules as finalized official OpenTT subsystems unless that status is explicitly announced by project maintainers.

When distributing such modules, clearly communicate maturity level and release status.

## 11. Requesting Permission

If you want to use OpenTT name/logo in a way not clearly covered above, request permission from project maintainers before publication.

Your request should include:
- intended use context,
- exact naming/branding mockup,
- distribution channel(s),
- whether the build is official, derivative, or commercial support around upstream.

## 12. Policy Evolution

This policy may evolve as the project grows.
Maintainers may clarify examples and edge cases to keep the policy practical and community-aligned.

## 13. Summary

OpenTT code is open and forkable under AGPL.
OpenTT identity (name/logo/official representation) must not be used in misleading ways.
Use, build, share, and contribute freely, while preserving transparency about who runs what.
